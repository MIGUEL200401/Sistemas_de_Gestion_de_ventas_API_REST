Sistema de Gestión de Ventas

Problema 4 Django + PostgreSQL DRF 8 Tablas 7 Relaciones

1. Descripción del Problema

Una empresa comercial requiere un sistema para administrar sus operaciones de ventas. El sistema debe gestionar clientes, productos, proveedores, pedidos, el detalle de los pedidos, facturas, pagos y las sucursales de la empresa.

La solución implementa una API REST construida con Django y Django REST Framework, conectada a una base de datos PostgreSQL organizada en un schema dedicado (ventas_schema). Se aplican las 13 características obligatorias de la guía: autenticación JWT, paginación, filtros, ordenamiento, soft delete, auditoría, Swagger, versionado de API, respuestas JSON estandarizadas, roles y permisos, nested serializers, exportación y logging.

4. Relaciones Implementadas

Producto → Proveedor

Tipo: N:1 (muchos a uno). Cada producto pertenece a un único proveedor. Un proveedor puede suministrar múltiples productos.

Pedido → Cliente

Tipo: N:1. Un cliente puede tener muchos pedidos. Cada pedido pertenece a un único cliente.

Pedido → Sucursal

Tipo: N:1. Cada pedido es atendido por una sucursal. Una sucursal puede gestionar múltiples pedidos.

DetallePedido → Pedido

Tipo: N:1. Un pedido tiene múltiples líneas de detalle. Cada detalle pertenece a un único pedido (CASCADE al eliminar).

DetallePedido → Producto

Tipo: N:1. Cada línea de detalle referencia un producto. El precio se guarda históricamente en el detalle.

Factura → Pedido

Tipo: 1:1. Cada pedido genera exactamente una factura. Implementada con UNIQUE en pedido_id.

Pago → Factura

Tipo: N:1. Una factura puede recibir múltiples pagos parciales. Cada pago referencia una única factura.

5. Justificación del Modelo

Normalización: el modelo está en 3FN. Los datos del proveedor no se repiten en productos; los datos del cliente no se repiten en pedidos. El precio unitario se almacena en detalle_pedidos (no en productos) para preservar el historial de ventas.

Soft Delete: todas las tablas implementan el campo activo en lugar de borrar registros físicamente, preservando la integridad histórica de facturas, pedidos y pagos.

Auditoría: los campos creado_por_id y modificado_por_id permiten rastrear qué usuario realizó cada operación.

Cálculos automáticos: el subtotal en detalle_pedidos y el total en pedidos/facturas se recalculan en cada operación de escritura mediante el método save() del modelo Django, garantizando consistencia.

Índices: se crean índices en todas las claves foráneas y en los campos usados frecuentemente en filtros (estado, activo, email) para optimizar el rendimiento de consultas.
